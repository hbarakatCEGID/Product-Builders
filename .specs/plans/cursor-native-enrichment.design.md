# Cursor-Native Rule Enrichment

## Problem

Template-generated .mdc rules are structurally correct but generic. They say "NEVER bypass authentication middleware" when they should say "Use `createClient()` from `src/lib/supabase/server.ts` in API routes — see `src/app/api/batches/route.ts` for the pattern."

The intelligence layer must be Cursor itself — no API LLMs, no external dependencies. The person setting up the product uses Cursor as their IDE.

## Design

### Architecture

```
product-builders analyze  →  analysis.json (heuristic data)
product-builders generate →  template .mdc rules (FALLBACK)
                           +  4 enrichment meta-rules

User opens Cursor         →  "Run enrich-critical"
                           →  Cursor reads template rules + codebase
                           →  Cursor rewrites .mdc files with depth
```

Templates provide structure and fallback. Cursor provides intelligence and specificity.

### Enrichment batches

4 batches, grouped by impact tier, ~5 min each:

| Batch | Meta-rule file | Rules enriched |
|-------|---------------|---------------|
| Critical | `enrich-critical.mdc` | database, auth-patterns, security |
| Architecture | `enrich-architecture.mdc` | architecture, coding-conventions, error-handling, api-patterns |
| Frontend | `enrich-frontend.mdc` | design-system, frontend-patterns, user-flows, state-and-config |
| Supporting | `enrich-supporting.mdc` | testing, performance, accessibility, i18n, cicd, git-workflow |

Batching prevents context window decay (research finding: rules generated later in long conversations have lower quality).

### Meta-rule structure

Each enrichment meta-rule contains:

1. **Heuristic context** — relevant analysis data for that batch's dimensions
2. **Gap-aware questions** — what heuristics couldn't determine (from `build_gap_aware_questions()`)
3. **Quality constraints** — cite files, show code, 1000-2500 words, don't hallucinate
4. **Per-rule instructions** — what files to analyze first, required sections, output format

Pattern per rule:
```
## Rule N: {rule-name}.mdc

Read `.cursor/rules/{rule-name}.mdc`, then rewrite it.

**Analyze these files first:**
- [specific paths derived from heuristic analysis]

**Required sections:**
- [section list with descriptions]

**Fill these gaps:**
- [gap-aware questions for this dimension]
```

### What gets generated by `product-builders generate`

```
profiles/{name}/
├── .cursor/rules/
│   ├── project-overview.mdc       ← template-generated (always)
│   ├── database.mdc               ← template-generated (fallback)
│   ├── auth-patterns.mdc          ← template-generated (fallback)
│   ├── ...
│   ├── contributor-guide.mdc      ← template-generated (always)
│   ├── enrich-critical.mdc        ← enrichment meta-rule (NEW)
│   ├── enrich-architecture.mdc    ← enrichment meta-rule (NEW)
│   ├── enrich-frontend.mdc        ← enrichment meta-rule (NEW)
│   └── enrich-supporting.mdc      ← enrichment meta-rule (NEW)
├── .cursor/hooks.json             ← template-generated (always)
├── .cursor/cli.json               ← template-generated (always)
└── docs/onboarding-*.md           ← template-generated (always)
```

### What stays template-based (no enrichment needed)

- `project-overview.mdc` — factual tech stack summary
- `contributor-guide.mdc` — role/zone permissions (deterministic)
- `hooks.json` — governance (deterministic zone × role logic)
- `cli.json` — governance (deterministic deny rules)
- `scopes.yaml` — zone definitions
- Onboarding guides — structural docs

### Enrichment meta-rule template

New Jinja2 template: `enrich-{batch}.mdc.j2`

Template context receives:
- `profile` — the full ProductProfile
- `gap_questions` — output of `build_gap_aware_questions()`
- `batch_rules` — list of rule names in this batch
- `rule_instructions` — per-rule analysis targets and required sections

### Cleanup after enrichment

After exporting to a product repo, enrichment meta-rules should NOT be committed:
- They contain analysis data (internal to platform team)
- They're only needed during setup, not day-to-day
- The `export` command should exclude `enrich-*.mdc` files

## Implementation

### Files to create

- `src/product_builders/generators/templates/enrich-critical.mdc.j2`
- `src/product_builders/generators/templates/enrich-architecture.mdc.j2`
- `src/product_builders/generators/templates/enrich-frontend.mdc.j2`
- `src/product_builders/generators/templates/enrich-supporting.mdc.j2`
- `src/product_builders/generators/enrichment.py` — batch definitions, per-rule instructions, enrichment generator class

### Files to modify

- `src/product_builders/generators/cursor_rules.py` — generate enrichment meta-rules alongside regular rules
- `src/product_builders/cli.py` — remove API-based `--enrich` flag (replaced by Cursor-native approach)

### Files to remove

- `src/product_builders/llm/client.py` — no longer needed (Cursor IS the LLM)
- `src/product_builders/llm/enrichment.py` — replaced by meta-rule templates
- `src/product_builders/llm/validator.py` — evidence validation stays in deep_analysis/schema.py
- `src/product_builders/llm/cache.py` — no caching needed (Cursor runs once)
- `src/product_builders/llm/config.py` — no model routing needed

### What stays

- All template improvements from Phase 1 (wired fields, new templates, frontmatter)
- Gap-aware deep analysis from Phase 2 (`build_gap_aware_questions()`)
- Evidence validation in `deep_analysis/schema.py`

4. Verify Cursor rewrites database.mdc, auth-patterns.mdc, security.mdc with project-specific content
5. Verify enriched rules reference actual file paths from the codebase
6. Verify `export` excludes enrich-*.mdc files
